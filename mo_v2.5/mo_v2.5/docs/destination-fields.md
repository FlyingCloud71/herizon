# `data/destinations.js` 字段说明

这份文件用于维护北京目的地样本数据。每一个目的地对象都不是普通旅游攻略，而是服务于一个问题：

> 腿脚不便、容易疲劳、需要随时休息和上厕所的妈妈，到这个地方能不能安心慢游？

所以字段要优先描述“体力压力、无障碍风险、休息和厕所、接驳、备用方案”，不要写成大众点评式推荐。

## 字段总览

```js
{
  id: "summer-palace",
  name: "颐和园",
  area: "海淀",
  cover: "湖边长廊、开阔水面、适合慢慢逛",
  halfDay: true,
  walkMinutes: 200,
  steps: 4,
  restroom: 150,
  seating: 4,
  transfer: 3,
  outdoorExposure: 5,
  crowdRisk: 4,
  nursingRoom: 1,
  bestTime: "工作日上午 9:00-11:30",
  dropoff: "建议从北宫门附近落客，减少入口外步行",
  restStops: ["北宫门游客服务点", "长廊沿线座椅", "昆明湖边阴凉处"],
  restroomTips: ["入园前先上厕所", "长廊前确认最近厕所位置"],
  backup: "如果当天人多或妈妈累了，只走北宫门到长廊一小段，拍照后直接返程。",
  caveat: "不建议上万寿山，台阶和坡道压力都偏高。",
  dataQuality: {
    status: "partiallyVerified",
    confidence: 3,
    lastChecked: "2026-05-09",
    notes: "说明哪些来自来源，哪些是产品化派生。",
    sources: ["https://example.com"]
  }
}
```

## 基础信息

### `id`

目的地唯一标识，用英文小写和连字符。

要求：
- 不要重复。
- 不要用中文。
- 页面跳转和数据查找会依赖它。

示例：

```js
id: "summer-palace"
id: "national-museum"
id: "beijing-fang"
```

### `name`

展示给用户看的目的地名称。

要求：
- 用最常见的中文名。
- 不要太长。
- 如果是街区，可以写“雍和宫周边”“北京坊”这种范围名。

### `area`

目的地所在城区或片区。

示例：

```js
area: "海淀"
area: "东城"
area: "朝阳"
```

### `cover`

一句话描述这个目的地为什么值得去，或它的慢游气质。

要求：
- 控制在 15-24 个中文字符左右。
- 不写营销词，比如“必去”“绝美”“网红”。
- 尽量写和慢游相关的信息。

好例子：

```js
cover: "室内、可控、厕所和休息点相对明确"
cover: "街区尺度小，适合吃饭、看店、短距离散步"
```

不建议：

```js
cover: "北京最热门的历史文化景点，风景优美"
```

## 慢游评估字段

这些字段会影响 `utils/evaluator.js` 里的可行性评分。

### `halfDay`

是否适合做成半日慢游。

当前 Demo 主要做半日慢游，所以大多数样本可以是 `true`。

填写建议：
- `true`：可以压缩成 2-4 小时，不需要完整游览也成立。
- `false`：目的地过大、交通过折腾，半日体验会很赶。

注意：目前代码还没有把 `halfDay` 纳入扣分，只是作为数据语义保留，后续可以扩展。

### `walkMinutes`

常规慢游安排的预计总步行时间，单位是分钟。

这里不是“景区完整逛完要多久”，而是本产品建议的半日慢游安排中，妈妈实际需要走路的总量。

填写口径：
- 只算步行，不算坐下休息、吃饭、看展停留。
- 以“行动偏慢的老人”为标准估计。
- 如果游览本身需要频繁进出、绕路、排队，也要适当加上体力消耗。

参考范围：

```text
10-25：非常轻松，适合体力弱或只想短坐短逛
30-40：可控，需要安排休息点
45-60：有压力，适合身体状态较好或游览范围必须收短
60+：不建议作为本 Demo 的主推慢游目的地
```

### `steps`

台阶、坡道、上下楼、路面连续性的压力评分，范围 `1-5`。

数值越高，风险越高。

评分口径：

```text
1：几乎无台阶，平路或电梯友好
2：少量台阶或轻微坡道，基本可绕开
3：有明显台阶、坡道或院落门槛，需要提醒
4：台阶/坡道较多，对腿脚不便者压力明显
5：大量台阶、爬坡、上下楼，不适合慢游主线
```

注意：
- 不要只看有没有无障碍通道，要看普通用户实际走起来会不会累。
- “可以绕行但绕行很远”的情况，仍然应给较高分。

### `restroom`

厕所数量，单位为“个”。评分时会结合 `walkMinutes` 计算每小时厕所数。

数值越高，越友好。它不是 `1-5` 级评分。

评分口径：

```text
restroomPerHour = restroom / (walkMinutes / 60)

低于 3 个/小时：密度偏低，对频繁如厕用户压力明显
3-4.9 个/小时：基本可用，但仍需要提前确认位置
5 个/小时及以上：密度相对友好
```

对于妈妈人群，厕所数量非常重要，但数量不等于距离。记录数量后还要考虑：
- 从入口到厕所是否远
- 是否需要排队
- 是否在同一层
- 是否容易问到和找到

### `nursingRoom`

育婴室或亲子卫生间数量，单位为“个”。

数值越高，越友好。它不是 `1-5` 级评分。

评分口径：

```text
0 个：没有明确可用的育婴室
1 个：有基础保障，但位置和排队仍需要提前确认
2 个及以上：数量更友好，可以加分
```

### `seating`

坐下休息便利度评分，范围 `1-5`。

数值越高，越友好。

评分口径：

```text
1：几乎没地方坐，只能站着等
2：座位少，休息不稳定
3：有一些休息点，但间隔较远
4：休息点较多，可边走边停
5：座椅、餐饮、室内空间充足，容易随时停下
```

可以算作休息点的地方：
- 公共座椅
- 游客中心
- 咖啡店/餐厅
- 展厅外长椅
- 商场公共区
- 阴凉处的固定座位

不建议把“路边台阶”“花坛边”当作可靠休息点。

### `transfer`

接驳便利度评分，范围 `1-5`。

数值越高，越方便。

评分口径：

```text
1：下车点远，打车/停车/地铁换乘都麻烦
2：能到达，但入口外步行较长
3：普通可达，需要选好入口
4：打车或公共交通比较方便
5：可以很接近目标点落客，返程也容易叫车
```

注意：
- 对老人来说，“地铁站很近”不一定等于方便，因为可能有换乘、楼梯、长通道。
- 优先考虑打车落点、入口距离、返程叫车难度。

### `outdoorExposure`

户外暴露度评分，范围 `1-5`。

数值越高，说明行程越依赖户外环境，遇到高温、暴晒、闷热、下雨、大风时越容易让妈妈疲劳。

这个字段不表示“当天热不热”，也不试图预测天气。它只描述目的地本身的结构特征：室内多不多、遮阴够不够、是否需要长时间在户外走、是否容易在户外排队。

评分口径：

```text
1：主要在室内，温度和休息节奏相对可控
2：室内外结合，户外只是一小段
3：有一定户外步行，但可以靠时间选择和休息点缓冲
4：户外时间明显，热天、晒天或雨天压力会变大
5：长时间暴露在户外，遮阴少或排队久，坏天气不建议
```

这个字段会和用户填写的“怕热或怕晒”联动。比如妈妈选择怕热时，`outdoorExposure >= 4` 会被当作风险点。

如果后续接入实时天气，可以把这个字段和天气数据组合：

```text
实际天气风险 = outdoorExposure × 当天温度/降雨/风力/空气质量
```

当前黑客松 Demo 暂时不做实时天气，所以这里只保留目的地自身的户外暴露度。

### `crowdRisk`

人流、排队、拥挤、噪音带来的风险评分，范围 `1-5`。

数值越高，风险越高。

评分口径：

```text
1：通常不拥挤
2：轻微人流，影响不大
3：普通热门程度，周末需注意
4：热门时段明显拥挤或排队
5：经常排队、人挤人，对老人体验影响很大
```

这个字段会和用户填写的“不喜欢人挤人”联动。

## 出行提示字段

这些字段会出现在评估页和“安心慢游卡”里，是 Demo 最打动人的部分。

### `bestTime`

推荐出行时间。

要求：
- 写具体时间段，不要只写“上午”。
- 体现避开人流、避开高温、配合老人作息。

好例子：

```js
bestTime: "工作日上午 9:00-11:30"
bestTime: "下午 15:00-17:30"
bestTime: "预约工作日上午场"
```

### `dropoff`

建议在哪里下车、从哪里进入、如何减少入口外步行。

要求：
- 从“少走路”的角度写。
- 尽量具体到入口、门、街区一侧。
- 如果地铁不适合老人，可以直接建议打车。

好例子：

```js
dropoff: "建议从北宫门附近落客，减少入口外步行"
dropoff: "打车到北京坊东侧入口，避开前门大街长距离步行"
```

### `restStops`

可靠休息点，数组格式。

要求：
- 建议 2-4 个。
- 写真实可理解的位置类型。
- 优先写“能坐下”的地方，而不是泛泛的“休息区”。

好例子：

```js
restStops: ["中庭休息区", "展厅外长椅", "文创店附近"]
restStops: ["书店座位", "街区公共座椅", "餐厅等位区"]
```

### `restroomTips`

厕所提醒，数组格式。

要求：
- 建议 2-3 条。
- 语气要像女儿提醒妈妈，不要像后台字段。
- 写“什么时候上厕所”“在哪里先确认”，不要只写“有厕所”。

好例子：

```js
restroomTips: ["进馆后先确认同层厕所", "离开展厅前再安排一次厕所"]
restroomTips: ["进书店或餐厅时先问厕所", "离开餐厅前安排一次厕所"]
```

### `backup`

备用方案，也就是妈妈累了、人多了、天气不好时怎么收缩游览范围。

要求：
- 必须具体。
- 要给出“立刻少走路”的替代动作。
- 语气温和，不要制造焦虑。

好例子：

```js
backup: "如果当天人多或妈妈累了，只走北宫门到长廊一小段，拍照后直接返程。"
backup: "如果排队太久，改为附近商场短坐和午饭，不硬等。"
```

不建议：

```js
backup: "视情况调整行程。"
```

### `caveat`

最重要的风险提醒。

要求：
- 只写一个最关键风险。
- 直接、具体、可行动。
- 不要写太多泛泛注意事项。

好例子：

```js
caveat: "不建议上万寿山，台阶和坡道压力都偏高。"
caveat: "需要提前预约，热门时段排队会增加疲劳。"
```

## 数据质量字段

### `dataQuality`

描述这条目的地数据的可靠程度。这个字段不直接参与当前 Demo 评分，但它决定数据能不能给真实用户使用。

建议结构：

```js
dataQuality: {
  status: "partiallyVerified",
  confidence: 3,
  lastChecked: "2026-05-09",
  notes: "北宫门、石舫、长廊方向可在官方信息中找到依据；休息点和厕所距离仍需现场核验。",
  sources: [
    "https://www.summerpalace-china.com/xnyh/index_mobile.htm"
  ]
}
```

### `dataQuality.status`

整体数据状态。

```text
verified：已用官方来源、地图和现场/照片核验，适合真实用户使用
partiallyVerified：有官方或高可信来源支撑，但慢游细节仍有派生成分
demoUnverified：只适合黑客松演示，不应直接给真实用户
needsReview：数据过旧、来源失效或存在明显冲突，需要重新核验
```

当前 Demo 大多数目的地都应该是 `partiallyVerified` 或 `demoUnverified`，不要为了好看写成 `verified`。

### `dataQuality.confidence`

慢游信息可信度，范围 `1-5`。

```text
1：演示假设，缺少可靠来源
2：有目的地基础信息，但厕所/座椅/落客点细节未核验
3：有官方或高可信来源，方向合理，但慢游细节仍需核验
4：官方信息 + 地图点位基本核验，可作为较可靠建议
5：官方信息 + 地图点位 + 现场/用户反馈核验，可给真实用户
```

### `dataQuality.lastChecked`

最后核验日期，格式用 `YYYY-MM-DD`。

注意：
- 目的地开放规则、入口、厕所、施工、人流都会变。
- 黑客松 Demo 可以用当天日期。
- 如果上线，建议 30-90 天重新核验热门目的地。

### `dataQuality.notes`

用一句话说明数据质量边界：

```text
哪些信息有来源支撑？
哪些内容是我们为了慢游做的派生？
哪些点还需要现场或地图核验？
```

### `dataQuality.sources`

来源链接数组。

优先级：

```text
1. 目的地官网/官方公众号/官方导览图
2. 政府文旅网站
3. 高德/百度/腾讯地图 POI 和室内图
4. 近期用户评价、照片、短视频
5. 队员现场核验记录
```

不要只放搜索结果页。尽量放能支持某个判断的具体页面。

## 填写原则

### 1. 先写“收短游览范围”，不要写“完整攻略”

本产品的安排不需要覆盖全部景点。对妈妈来说，逛得少但舒服，比打卡更多更重要。

### 2. 每个目的地都要有退出机制

`backup` 是核心字段，不是附属字段。它体现产品真正理解老人出游：随时可以改、随时可以停。

### 3. 评分要保守

如果不确定，就往风险更高的方向填。黑客松 Demo 里，保守判断比盲目推荐更可信。

### 4. 文案要生活化

用户不是专业导游，而是想带妈妈出门的女儿。文案应该像“我已经替你想好了”，不是像景区公告。

### 5. 避免承诺绝对安全

不要写：

```text
完全无障碍
一定不累
保证有座位
随时都有厕所
```

建议写：

```text
相对友好
建议提前确认
优先选择
尽量减少
```

## 和评分逻辑的关系

当前评分逻辑在 `utils/evaluator.js`：

- `walkMinutes` 超过妈妈填写的步行上限会扣分。
- 如果妈妈选择“尽量不走台阶”，`steps >= 3` 会扣分。
- 如果妈妈选择“需要频繁上厕所”，`restroomPerHour < 3` 会扣分。
- 如果妈妈选择“需要随时能坐下”，`seating < 4` 会扣分。
- 如果妈妈怕热，`outdoorExposure >= 4` 会扣分。
- 如果妈妈不喜欢人挤人，`crowdRisk >= 4` 会扣分。
- 如果妈妈需要轮椅/助行器，`steps >= 3` 会额外扣分。
- 如果妈妈需要育婴室，`nursingRoom <= 0` 会扣分，`nursingRoom >= 2` 会加分。

评估结果：

```text
68 分及以上：推荐
48-67 分：谨慎推荐
47 分及以下：不建议
```

所以维护数据时要特别注意这几个阈值：

```text
steps >= 3：对避台阶用户会变成风险点
restroomPerHour < 3：对厕所敏感用户会变成风险点
seating < 4：对需要休息用户会变成风险点
outdoorExposure >= 4：对怕热用户会变成风险点
crowdRisk >= 4：对避人流用户会变成风险点
nursingRoom <= 0：对需要育婴室用户会变成风险点
nursingRoom >= 2：对需要育婴室用户会加分
```
